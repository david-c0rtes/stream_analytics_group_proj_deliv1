# Madrid Food Delivery Streaming Simulation

## Milestone 1: Streaming Data Feed Design & Generation

---

## 1. Project Overview

In this project we built a synthetic streaming data generator that simulates how a real food delivery platform operates. The system is inspired by real-world marketplaces such as Uber Eats, Deliveroo, and Glovo, where thousands of operational events are constantly generated by different services.

Real-time food delivery platforms generate large-scale event streams including:

- **Order lifecycle events** — Creation, confirmation, preparation, pickup, and delivery
- **Courier operational signals** — Availability, assignments, arrivals, and delivery completion
- **Restaurant preparation updates** — Status transitions from accepted to ready-for-pickup

For this milestone, our goal was to design and simulate these streams so they can later be used for testing real-time analytics pipelines.

Specifically, in Milestone 1 we focused on:
- Designing the event feeds
- Defining the event schemas
- Implementing a synthetic generator that produces realistic event streams
- Injecting edge cases commonly found in real streaming systems
- The result is a simulator capable of producing schema-compliant streaming data that mimics how a real delivery platform behaves.

## 2. Execution Modes

The system supports two execution modes.

### Mode 1 — Standard Run (Batch Generation)

In this mode the simulator generates a full dataset as quickly as possible. This is useful when we want to quickly generate data for testing or offline analysis.
During execution:
- Events are generated at maximum speed
- Data is written to NDJSON and AVRO files
- Output directories are cleaned before each run to ensure the dataset belongs to a single simulation

Run with:
```bash
python3 run_simulation.py
```

### Mode 2 — Live Simulation Mode

In this mode the simulator behaves more like a real streaming environment. Instead of generating events instantly, the system:
- Preserves generation order (including out-of-order arrivals)
- Writes them incrementally to files
- Optionally replays events with time delays

This creates a stream that behaves similarly to a real production event feed.

```bash
python3 run_simulation.py --live
```

To replay strictly by event time for demos:
```bash
python3 run_simulation.py --live --live-sort-by-event-time
```

## 3. System Concept

To make the simulation realistic, we modeled the basic workflow of a food delivery marketplace.
1. A customer places an order
2. A restaurant accepts and prepares it
3. A courier is assigned
4. The courier picks up the order
5. The courier delivers the order

The high-level event flow is:

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│ Customer places  │────▶│ Restaurant      │────▶│ Courier         │
│ order (created)  │     │ accepts &       │     │ assigned        │
└─────────────────┘     │ prepares       │     └────────┬────────┘
                         └────────┬───────┘              │
                                  │                       │
                                  ▼                       ▼
                         ┌─────────────────┐     ┌─────────────────┐
                         │ Restaurant      │     │ Courier picks   │
                         │ marks ready     │────▶│ up food         │
                         └─────────────────┘     └────────┬────────┘
                                                           │
                                                           ▼
                                                  ┌─────────────────┐
                                                  │ Courier         │
                                                  │ delivers order  │
                                                  └─────────────────┘
```

Each step emits events that are published to distinct feeds. The system interleaves events from both feeds by `event_time`, simulating how a real streaming platform would deliver events to downstream consumers.

## 4. Streaming Feed Design (Milestone 1.1)

To model the platform realistically, we designed two main streaming feeds that represent the core operations of the system.

Separating the feeds reflects how real production systems work, where different services produce independent streams that are later joined in analytics pipelines.

### Feed 1: Order Lifecycle Feed

**What it represents:** The complete lifecycle of a food delivery order from creation to delivery or cancellation.

**Why it is essential:** Orders are the primary business entity. Every revenue event and customer interaction is tied to an order. The order lifecycle feed provides a single source of truth for order state transitions.

**Analytics enabled:**


| Analytics Domain      | Use Case                                                  |
| --------------------- | --------------------------------------------------------- |
| Throughput            | Orders per hour, peak detection, capacity planning        |
| Preparation time      | Prep time monitoring, kitchen SLA, restaurant performance |
| Delivery SLA          | End-to-end fulfillment time, on-time delivery rates       |
| Cancellation behavior | Cancellation by user/restaurant/system, pattern analysis  |


### Feed 2: Courier Operations Feed

This feed represents the operational behavior of couriers, including availability, assignments, and delivery completion.

Courier events allow us to model the supply side of the marketplace.

Using this feed we can analyze:

**Analytics enabled:**


| Analytics Domain       | Use Case                                           |
| ---------------------- | -------------------------------------------------- |
| Availability           | Online courier count by zone, shift coverage       |
| Dispatch efficiency    | Time-to-assignment, assignment rate during peaks   |
| Route performance      | Travel time to restaurant, travel time to customer |
| Operational congestion | Zone-level pickup delays, abandonment patterns     |


Separating these feeds allows downstream pipelines to subscribe only to the events they need and to perform stream-stream joins on `order_id` and `courier_id` when both perspectives are required.

## 5. Event Schema Design (Milestone 1.2)

Both feeds are defined using **AVRO schemas** stored in `.avsc` files. AVRO was chosen for the following reasons:

- **Schema evolution support** — Forward and backward compatibility for adding or deprecating fields
- **Compact binary serialization** — Efficient storage and transmission in streaming pipelines
- **Compatibility with streaming systems** — Native support in Kafka, Event Hubs, and Spark Structured Streaming
- **Strong typing** — Enums and nullable types enforce structure at serialization time

### Key Schema Features

- **Enums for event types** — Each feed uses an enum (`OrderEventType`, `CourierEventType`) to restrict `event_type` to a fixed set of values
- **Optional fields** — `courier_id`, `order_id`, `restaurant_id`, and time estimates use `["null", "T"]` for partial events
- **Versioned schema design** — `event_version` (default `"1.0"`) supports evolution; namespaces (`com.madrid.delivery.order.v1`) enable versioned types
- **Nullable fields** — `actual_prep_time`, `actual_delivery_time`, `offline_reason` etc. allow events to be emitted before values are known

### Output Formats

Events are produced in two formats:


| Format            | Purpose                                                                           |
| ----------------- | --------------------------------------------------------------------------------- |
| **JSON (NDJSON)** | Human-readable inspection, debugging, and integration with tools that expect text |
| **AVRO**          | Efficient binary serialization for streaming ingestion and storage                |


JSON output includes `event_time_epoch_ms` and `ingestion_time_epoch_ms` for compatibility with Event Hubs and Spark Structured Streaming. AVRO schemas store timestamps as `long` (epoch milliseconds).

## 6. Event-Time and Join Design

### Timestamps

Each event includes two timestamps required for event-time analytics:


| Field              | Semantics                               |
| ------------------ | --------------------------------------- |
| **event_time**     | When the business event occurred (UTC)  |
| **ingestion_time** | When the event entered the stream (UTC) |


**Difference:** `event_time` reflects the logical occurrence of the event in the simulated world. `ingestion_time` models when the event would be received by a streaming system. These can differ when late events are simulated (network delay, retries).

**Why event-time is necessary:**

- **Windowed analytics** — Tumbling, sliding, and session windows must be computed on `event_time` to produce correct results
- **Late event handling** — Events may arrive after their logical time; watermarks and allowed lateness depend on event-time semantics
- **Watermark-based processing** — Streaming engines use event-time to determine when windows can be closed

### Join Identifiers

The following identifiers enable stream-stream and stream-table joins:


| Identifier                                              | Scope      | Join use case                                 |
| ------------------------------------------------------- | ---------- | --------------------------------------------- |
| `order_id`                                              | Both feeds | Join order lifecycle with courier operations  |
| `courier_id`                                            | Both feeds | Enrich order events with courier attributes   |
| `restaurant_id`                                         | Order feed | Enrich with restaurant metadata               |
| `user_zone_id`, `restaurant_zone_id`, `courier_zone_id` | Both feeds | Zone-level analytics, demand-supply imbalance |

## 7. Realistic Simulation Model (Milestone 1.3)

The generator produces realistic operational behavior through configurable demand distributions and external factors.

### Demand Distributions

- **Lunch peak** — 12:00–14:30, configurable multiplier range (default 2.0×–3.0×)
- **Dinner peak** — 19:30–22:00, configurable multiplier range (default 1.8×–2.6×)
- **Weekday vs weekend** — Weekend demand multiplier (default 1.1×–1.5×)
- **Zone-level differences** — Spawn probability per zone derived from population data

Demand uses a smooth time-of-day curve (Gaussian bumps) rather than step changes, producing gradual ramps into and out of peaks.

### Restaurant and Category Effects

Restaurant demand varies by:

- **Category** — burger, pizza, mexican, sushi, etc., with category-specific multipliers
- **Location** — Zone-based spawn probability from Madrid population data
- **Promotions** — Recurring (e.g., Taco Tuesday) and random flash promotions
- **Football matches** — Demand surges in stadium zones (Chamartín, San Blas-Canillejas) for match-day categories

### Configuration Parameters


| Parameter                  | Description                       |
| -------------------------- | --------------------------------- |
| Restaurant count           | Number of restaurants in the pool |
| Courier count              | Number of couriers                |
| Initial online fraction    | Share of couriers online at start |
| Demand surge multipliers   | Lunch, dinner, weekend, holiday   |
| Promotion periods          | Recurring and random flash        |
| Cancellation probabilities | User, restaurant, courier, system |

## 8. Geographic Model

The simulation uses **Madrid** as the geographic model. The city is divided into **21 operational zones** corresponding to official districts (e.g., Chamartín, San Blas-Canillejas, Retiro).

The zone system is used for:

- **Customer locations** — Zone-based spawn probability from population
- **Restaurant locations** — Restaurants are assigned to zones; each has category and prep-time parameters
- **Courier routing** — Couriers are associated with zones; assignment considers zone proximity
- **Delivery travel times** — A zone-to-zone travel matrix defines mean travel time, standard deviation, zone hops, delay probability, and cancellation sensitivity

The travel matrix defines estimated delivery times and routing difficulty between every zone pair, enabling realistic travel time sampling and cancellation modeling based on zone hops.

## 9. Streaming Edge Cases (Milestone 1.3)

The generator deliberately produces edge cases required to test streaming system behavior. Configurable probabilities control the frequency of each.


| Edge Case                        | Description                                                   | Implementation                                                           | Testing purpose                                |
| -------------------------------- | ------------------------------------------------------------- | ------------------------------------------------------------------------ | ---------------------------------------------- |
| **Out-of-order events**          | Events arrive in an order different from their `event_time`   | Configurable probability; events may be emitted in non-monotonic order   | Watermark behavior, late-data handling         |
| **Late events**                  | Events with `ingestion_time` significantly after `event_time` | `is_late_event: true`, `event_delay_seconds` set; ingestion time delayed | Allowed lateness, speculative vs final results |
| **Duplicate events**             | Same logical event ingested more than once                    | Duplicate emitted with new `event_id`                                    | Idempotency, deduplication logic               |
| **Missing lifecycle steps**      | `order_delivered` emitted without prior `order_picked_up`     | Configurable `missing_step_probability` skips pickup event in some flows  | Robust state handling for incomplete sequences |
| **Impossible durations**         | Anomalous prep or travel times                                | Anomaly flag and configurable injection probability                      | Anomaly detection pipelines                    |
| **Courier offline mid-delivery** | Courier goes offline during an active delivery                | `courier_offline_mid_delivery` event with `offline_reason`               | Failure handling, reassignment logic           |


These edge cases are necessary to validate:

- **Watermarks** — Correct handling when events arrive late
- **Event-time windows** — Correct aggregation when order differs from processing time
- **Fault tolerance** — Idempotent sinks and robust join behavior

## 12. Assumptions


| Domain                          | Assumption                                                                                                     |
| ------------------------------- | -------------------------------------------------------------------------------------------------------------- |

## 15. Milestone 2 Compliance (Spark + Kafka/Event Hub)

This section maps the professor requirements to the current implementation.

### Requirement Mapping

| Professor requirement | Implementation in this repo |
| --- | --- |
| Spark consumes Event Hub through Kafka-compatible endpoint | `spark_streaming_job.py` uses `spark.readStream.format("kafka")` and subscribes to Event Hub names from environment variables |
| `kafka.bootstrap.servers = {namespace}.servicebus.windows.net:9093` | Built as `KAFKA_BOOTSTRAP` from `EVENTHUB_CONN_STR` in `spark_streaming_job.py` |
| `kafka.sasl.mechanism = PLAIN` | Set in `kafka_opts` in `spark_streaming_job.py` |
| `kafka.security.protocol = SASL_SSL` | Set in `kafka_opts` in `spark_streaming_job.py` |
| JAAS string with username `$ConnectionString` and full Event Hub connection string as password | Set in `KAFKA_SASL` and passed to `kafka.sasl.jaas.config` in `spark_streaming_job.py` |
| Producer publishes AVRO data | `event_publisher.py` supports AVRO payloads when `EVENTHUB_PAYLOAD_FORMAT=avro` (JSON remains supported for compatibility) |
| Spark Structured Streaming persists to Parquet | `spark_streaming_job.py` writes micro-batches to Blob as Parquet with `foreachBatch` |

### Runbook (End-to-End)

1. Install dependencies:
   ```bash
   # One-command full environment:
   python3 -m pip install -r requirements_all.txt

   # Or install only the layer you need:
   python3 -m pip install -r requirements.txt
   python3 -m pip install -r requirements_spark.txt
   python3 -m pip install -r requirements_dashboard.txt
   ```
2. Configure `.env` with:
   - `EVENTHUB_CONN_STR`
   - `EVENTHUB_ORDERS`
   - `EVENTHUB_COURIERS`
   - `AZURE_STORAGE_CONNECTION_STRING`
   - `AZURE_STORAGE_CONTAINER` (optional, defaults to `dashboard-data`)
3. Terminal A (Spark consumer):
   ```bash
   python3 spark_streaming_job.py
   ```
4. Terminal B (simulator producer in AVRO mode):
   ```bash
   python3 run_simulation.py --config simulation_config_live.yaml --live --eventhub-payload-format avro
   ```
5. Terminal C (dashboard):
   ```bash
   streamlit run streamlit_app.py
   ```

### Validation Evidence Checklist

- Spark startup logs show the Event Hub Kafka endpoint and stream startup.
- Spark logs show periodic Blob writes: `order_lifecycle_events_spark_*.parquet` and `courier_operations_events_spark_*.parquet`.
- Streamlit dashboard updates while simulation is running (Blob-first; local fallback if Blob unavailable).
- If needed for backward compatibility tests, rerun simulator with `--eventhub-payload-format json`.

  - `run_simulation.py`
